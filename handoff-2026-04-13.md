# Handoff — 2026-04-13

## Session Summary

This session built the Tasks panel feature end-to-end and then replaced its persistence layer. The Tasks panel was first built with electron-store persistence (multi-list model with project tabs), then the entire data layer was rewritten to use `project-implementation.md` as the single source of truth — a markdown file in the terminal's CWD that enables two-way sync with Claude Code and external editors. Font zoom was also extended: Cmd+/- now scales the compose bar alongside the terminal, and Cmd+Option+/- scales all app fonts. CSV import was added with a drag-and-drop modal. All changes are committed, pushed, and a DMG was built.

## Key technical decisions

### MD file as source of truth instead of electron-store
The original Tasks panel used electron-store (JSON blob). This was replaced with `project-implementation.md` in the project directory. **Why:** Claude Code can directly read and edit this file, enabling two-way sync. A user can update task status in the terminal via Claude Code and see it reflected in the Tasks panel. The MD format is human-readable and version-controllable. **Tradeoff:** More complex persistence (parser/writer/watcher) vs. simple JSON. Worth it for the workflow integration.

### Removed multi-list tabs
The original design had "Project 1", "Project 2" tabs within the Tasks panel for organizing tasks into lists. **Why removed:** With MD file persistence, each project directory has its own `project-implementation.md`. The "project" is the directory itself, not a user-created tab. Multiple lists within one file would complicate the format. **Impact:** Simpler UI, simpler data model, but if the user wants separate task lists in the same directory they can't.

### CWD polling with ref (not state) for comparison
The CWD poll uses `cwdRef.current` instead of `cwd` state for the `!==` comparison inside `setInterval`. **Why:** React state in an interval closure captures the value at effect creation time (stale closure). A ref always has the latest value. This was the root cause of the initial bug where tasks weren't loading after `cd`. **Impact:** The `useEffect` dependency array doesn't include `cwd`, which is intentional.

### Atomic writes via temp file + rename
Task writes go to `project-implementation.md.tmp` first, then `fs.rename()` to the final path. **Why:** If the process crashes mid-write, the original file is intact. `rename()` on the same filesystem is atomic on macOS/Linux.

### tasksWriteInProgress flag with 500ms timeout
After writing, the flag stays true for 500ms to let the chokidar event pass through without triggering a re-parse. **Why:** Without this, every save would trigger the file watcher which would re-parse and re-set state, creating a feedback loop and potential data loss if the re-parse happens before the write completes.

### TaskComment.timestamp changed from number to string
Was `timestamp: number` (Unix ms), now `timestamp: string` ("YYYY-MM-DD HH:MM"). **Why:** The MD format uses human-readable timestamps. Converting back and forth between Unix timestamps and strings would lose timezone info and add complexity. String timestamps are directly writable/parseable from the MD format.

### Feedback as separate concept from comments
The spec required both `### Feedback` and `### Comments` sections per task. **Why separate:** Comments are general notes/log entries. Feedback is review input from stakeholders or AI. They serve different purposes and the user may want to include one but not the other when sending to terminal.

## Deferred / not built

- **Image storage in MD as markdown image syntax** — The spec mentioned storing images as `![](/.flowt/image.png)` references in the task body. Currently images are still stored as base64 data URLs in the Task object's `images[]` array (in memory only, not persisted to the MD file). This should be implemented when image persistence to the MD format is needed.
- **Spec field in header** — The `> Spec:` field in the MD header defaults to empty. No UI to set it. Could add an editable field in the Tasks panel header.
- **Task reorder persistence** — Reordering within the UI works but the MD file order is based on the order of task blocks in the file. Reordering updates the in-memory `order` field but the writer outputs tasks in the order they appear in the `tasks` array, which may not match the visual reorder perfectly across status filters.
- **Error indicator on individual cards** — The spec mentioned showing a subtle error indicator on the affected card if a write fails. Currently write failures are logged to verbose but not surfaced on the card.
