## Key technical decisions

### Why `claude --system-prompt` instead of injecting system prompt as first message

Anthropic's `claude` CLI supports `--system-prompt <prompt>` (verified Q1 via `claude --help`). Passing as a separate flag avoids burning a turn on a pre-message and keeps the conversation log clean. **Tradeoff rejected:** sending as first message would have wasted ~500 tokens and obscured the conversation start.

### Why one-shot `claude -p --resume` (not interactive TUI)

Each advisor turn is a one-shot `child_process.spawn('claude', ['-p', msg, '--resume', sid, '--output-format', 'json', ...])`. Anthropic's session storage at `~/.claude/projects/<cwd>/` handles continuity. **Tradeoff rejected:** running `claude` interactively in a hidden node-pty would mean parsing the full TUI (alternate screen buffer, ANSI codes, redraws) — fragile and version-coupled. Headless mode gives clean JSON stdout. Cost: each turn is a fresh process spawn (~2-5s overhead per turn). Acceptable.

### Why consent-based attribution (not PATH shim)

The PATH shim approach (option D) was implementable cleanly via `<bundle>/Resources/shims/claude` + OSC 1337 escape sequence for tab attribution. **Rejected** because the shim has multiple silent bypass paths: absolute path invocation (`/usr/local/bin/claude`), user-defined `claude` shell function, alias conflicts. All three produce the same symptom: SendToAdvisorButton sits disabled with no clear reason. Founders can't diagnose without knowing about shim internals. Consent-based observation has visible failure modes (Restart CC banner, hooks-removed banner, diagnosable toasts) at the cost of one-time founder action (consent + restart CC after scaffold). **Worth it for diagnosability.**

### Why two-env-var hook gating (`FLOWT_ADVISOR_PROCESS=1` + no gate on Stop hook)

PreToolUse hook (`sandbox-check.sh`) is gated by `FLOWT_ADVISOR_PROCESS=1` — Flowt sets this only when spawning the advisor. Executor CC and any other CC sessions in the project pass through the hook with exit 0 (no enforcement). **Stop hook (`cc-stop-hook.sh`) has NO env gate** — fires for any CC in the project. This is intentional: Flowt no longer spawns the executor (option D rejected), so we can't tag it. Stop hook captures every turn from any CC; chokidar attribution figures out which tab. **Tradeoff:** if founder runs CC outside Flowt in the same project directory (e.g., from Terminal.app), Stop hook fires and writes `cc-turns/*.json` even though Flowt isn't watching. Files accumulate harmlessly; founder can `rm -rf .flowt/cc-turns/` at any time. v1.1 may add garbage collection.

### Why HTML comment delimiters for drafts

Locked: `<!-- FLOWT_DRAFT_START -->` / `<!-- FLOWT_DRAFT_END -->`. **Tradeoff rejected:** triple-pipe fences (`|||`) would conflict with markdown table cells. HTML comments survive markdown rendering (don't display as visible text in the chat) and don't conflict with code blocks inside the draft. The advisor's system prompt footer instructs verbatim usage of these markers; renderer parses inner content into a draft card.

### Why two send paths (compose-bar + direct PTY) instead of one

The original spec stated "the Enter key is the human gate, always." The locked-and-shipped design adds a secondary "Send and execute" button that bypasses compose-bar review. **Reasoning for not violating the principle:** auto-send (rejected) is "send after 5 seconds without intervention." Direct send is "click an explicit secondary button." Both are human gates; the second is just one click instead of two. **Mitigations:** events are logged with `mode: 'compose' | 'direct'` so retrospective drift detection is possible (analytics deferred to v1.1+ if drift becomes real); visual hierarchy in the card (primary green vs secondary outline) keeps the safe path the eye-default; no auto-confirmation; founder must explicitly click. Locked into v1 per founder's request and reasoning.

### Why per-CWD advisor scoping (not per-tab, not per-window)

Multiple terminal tabs in the same project share one advisor instance — matches the existing Tasks panel pattern (per CLAUDE.md). Switching tabs to a different CWD swaps the advisor panel content. **Tradeoff rejected:** per-tab would mean two tabs in the same project have diverged advisor contexts (wasteful); per-window would ignore which project the founder is working on (advisor's CLAUDE.md context would be wrong half the time).

### Why a footer is non-overrideable while body is

The advisor system prompt has two parts: a body (replaceable by `<cwd>/.flowt/advisor.md` per Phase 6 lock) and a footer (always appended by Flowt). **The footer contains the draft delimiter mechanics** (`<!-- FLOWT_DRAFT_START -->` instructions). If founders could override the footer, they could accidentally break the Draft Message to CC button by writing a custom advisor.md without the delimiter rules. Founder owns persona; Flowt guarantees mechanics. **Same pattern as how Claude Code itself layers internal directives over user CLAUDE.md.**

### Why `logAdvisorError` takes eventType as parameter (not hardcoded)

The first draft of the helper signature hardcoded `error_turn`. **Caught during Phase 9 §9.6 review** — would silently misclassify all errors and break post-hoc jq queries (`select(.type == "extract_failed")` would return empty). Locked signature accepts `eventType: AdvisorErrorEventType` union as a parameter, with the call site responsible for passing the correct type. The TypeScript union prevents typos at compile time.

### Why `PendingDraft.state` distinguishes `DRAFT_SENT_COMPOSE` and `DRAFT_SENT_DIRECT`

Originally a single `DRAFT_SENT` state. **Caught during Phase 5 review** — the `send_to_terminal` event records `mode: 'compose' | 'direct'` for analytics; the in-memory state needs to match so the type system can enforce the distinction. The two states are also useful for any future per-mode UI animation differentiation.

### Why synthetic system bubble derived from `draft_cc_clicked` event (not a new event type)

When the founder clicks Draft Message to CC, the chat shows a dimmed system bubble: `Asking advisor to draft the executor message…`. **Locked decision** in Step 3 of /project-planner: the bubble is NOT logged as its own event type. Instead, `replayEvents` (the pure replayer function) derives the synthetic bubble from the presence of `draft_cc_clicked` event without a subsequent `draft_produced`/`draft_parse_fail`. **Why:** keeps the event log lean (events represent founder/advisor actions, not UI ephemera); a single source of truth (the log) drives all chat structure. Both task-008 and task-038 are designed consistently around this rule.

### Why Phase 4 (resilience) is last, not interleaved

Could have shipped error UX as each happy path lands. **Rejected** because speculative error UX risks designing for failure modes that don't actually occur in practice. By P4, dev experience during P1-P3 will have surfaced real error categories. **Critical clarification (per founder):** errors are still LOGGED via `logAdvisorError` from P1 onward — only the visible UX layer (banners, toasts, recovery flows) is deferred to P4. Console.error gives visibility during dev; data capture is complete from day one; P4 just adds the UX skin on top.

### Why /copy clipboard fallback is deferred (not built in v1)

Spec originally planned both Stop hook (primary) and `/copy` clipboard fallback (when hook missing). **Cut for v1** — consent flow guarantees hooks are installed; Stop hook fires for any CC session in the project; missing-file case is genuinely rare (founder ran CC pre-consent without restarting; hook script error). Diagnosable error toast pushes founder to recovery action: `No CC turn captured. If you started Claude Code before enabling the advisor, restart it (Ctrl+C, then claude) to enable Send to Advisor.` The fallback infrastructure (PTY write blocking, clipboard polling, menu-confirm timing) is real implementation cost. Threshold for re-introducing: `extract_failed` events > ~3/week per active founder.

### Why hardened task-024 (P1 manual test) with 12 sub-steps

The P1 success criterion #2 ("Hooks load") depends on multiple components working in concert: scaffolder wrote settings.json correctly, cc-stop-hook.sh got copied + chmod'd, chokidar watcher set up correctly, IPC event chain wired, banner state flipped. **Caught during planning Step 4 final review** — without explicit sub-steps, a P1 build could pass the high-level "banner dismissed" test by accident (banner was never shown due to a different bug) or fail subtly (banner shows but dismisses on tab switch, not on hook fire). Sub-steps 10a through 10l in task-024 catch each link in the chain individually.
